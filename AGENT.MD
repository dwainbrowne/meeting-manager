# Agent Rules for Meeting Manager

This repository is public. Treat every change as if it will be visible to customers, partners, and the internet.

## Non-Negotiable Credential Rules

1. Never commit API keys, access tokens, passwords, private keys, session cookies, database URLs, webhook secrets, service account files, Cloudflare tokens, OpenAI keys, GitHub tokens, or provider credentials.
2. Never paste real secret values into source code, documentation, examples, tests, screenshots, logs, issue text, pull request text, or commit messages.
3. Keep environment variables local. Use `.env`, `.env.local`, `.dev.vars`, local keychains, or managed secret stores such as Cloudflare secrets. These files must not be committed.
4. Public examples may show variable names only, such as `OPENAI_API_KEY`, `CLOUDFLARE_API_TOKEN`, or `DATABASE_URL`. Do not include real values.
5. If a secret is found in the repo, stop feature work immediately. Remove it from the working tree, rotate the credential, and document only the remediation, not the secret.

## Required Check Before Every Improvement

Before every upgrade, feature, refactor, dependency change, infrastructure change, commit, or push:

1. Check the changed files for credentials and private values.
2. Run a secret-pattern scan across the repository.
3. Confirm `.env*`, `.dev.vars*`, `.wrangler/`, Terraform state, local uploads, build output, and dependency folders are ignored.
4. Confirm no generated logs or screenshots contain private tokens, URLs with credentials, account identifiers that should stay private, or local environment values.
5. If Cloudflare, AI providers, databases, email, calendar, GitHub, or other infrastructure APIs are touched, do an extra pass for provider-specific credentials.

## Cloudflare and Infrastructure Rules

1. Cloudflare API tokens must live only in the local environment, Cloudflare dashboard, or Cloudflare secret storage.
2. Do not commit `.dev.vars`, `.wrangler`, Terraform state, or files generated by deployment tooling that include account details or credentials.
3. Prefer least-privilege Cloudflare tokens scoped only to the resources needed.
4. Keep public configuration separate from secret values. Commit configuration only when it is safe to share.
5. Document setup steps in plain language without exposing private account values.

## Product Direction

Build Meeting Manager as a clear, practical workspace for meeting history, notes, summaries, transcripts, decisions, and follow-up tasks. The value is helping people answer:

- What happened?
- What did we decide?
- Who owns the next step?
- What is overdue?
- What did we discuss with this person or company before?

Prioritize infrastructure and data safety first, then move the app from local sample data to production-ready APIs and storage, then improve the daily meeting workflow.

# Meeting Manager App PRD + Data Structure
1. Product Overview

The app is a meeting workspace that organizes companies, contacts, meetings, notes, transcripts, AI summaries, and action items in one place.

The core layout has four main areas:

Top bar
Search, filters, AI assistant, and new meeting creation.
Left panel
List of companies and people. Users can switch between All, Business, and Personal.
Middle meeting list
Shows meetings related to the selected company or person.
Main meeting detail area
Shows meeting title, attendees, date, summary, notes, transcript, and AI actions.
Right action panel
Shows open and completed action items grouped by assignee.

4. PRD: Meeting Manager System
Product Name

Minute — Meeting Workspace

Product Goal

Build a workspace where users can manage business and personal meetings by company, contact, meeting history, AI-generated summaries, transcripts, notes, decisions, and action items.

The product should make it easy to answer:

“What happened in this meeting?”
“What needs to be done?”
“Who owns it?”
“What is overdue?”
“What did we discuss with this company before?”

5. Target Users
Primary User

A business owner, account manager, operator, consultant, or team lead who has many calls across different companies and people.

Secondary Users

Team members who need visibility into meeting outcomes, action items, and follow-ups.

External Contacts

People who attend meetings but may not have access to the full app.

6. Core User Stories
Company and Contact Management

As a user, I want to see a list of companies and contacts on the left side so I can quickly navigate between relationships.

As a user, I want to filter contacts by All, Business, and Personal so I can separate work-related meetings from personal ones.

As a user, I want each company to show its industry, contact count, meeting count, and open action count so I can understand activity at a glance.

Meeting Management

As a user, I want to click a company or person and see all related meetings.

As a user, I want to click a meeting and view the summary, notes, transcript, attendees, and action items.

As a user, I want to create a new meeting manually.

As a user, I want to paste or upload meeting notes and convert them into a structured meeting record.

AI Summarization

As a user, I want to summarize a meeting with AI so I can quickly extract the key points.

As a user, I want AI to extract action items, decisions, risks, blockers, and follow-up questions from the transcript.

As a user, I want to review and edit AI-generated content before treating it as final.

Action Item Management

As a user, I want action items grouped by assignee so I know who owns what.

As a user, I want to see open, completed, overdue, and upcoming action items.

As a user, I want to mark an action item as complete.

As a user, I want to assign action items to internal users or external contacts.

Search and Filters

As a user, I want to search across companies, people, meetings, notes, transcripts, and action items.

As a user, I want filters for company, person, meeting status, date range, action status, and priority.
